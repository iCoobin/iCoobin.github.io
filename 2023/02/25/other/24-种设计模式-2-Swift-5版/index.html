24 种设计模式-2 (Swift 5版)

结构型模式（Structural Patterns）

1. 适配器模式（Adapter Pattern）

适配器模式（Adapter Pattern）是一种结构型设计模式，用于将一个类的接口转换成客户端所期望的另一种接口。它允许两个不兼容的接口能够一起工作。

适配器模式常用来解决以下问题：

  1. 系统需要复用一个已存在的类，但是其接口与系统所需要的接口不兼容。
  2. 不想修改现有的接口，但是需要创建一个新的类能够调用现有类的接口。
  3. 两个不兼容的类需要协同工作，但是不能直接修改它们的接口。

在适配器模式中，适配器充当中间人角色，它接收来自客户端的请求并将其转换为被适配对象的请求。适配器模式可以分为两种类型：对象适配器和类适配器。对象适配器使用组合的方式连接适配器和被适配对象，类适配器使用继承的方式连接适配器和被适配对象。

以下是一个使用对象适配器的 Swift 5 代码示例，用于将一个高级音乐播放器的接口适配成一个简单音乐播放器的接口：

  • Swift Code

    // 被适配的高级音乐播放器
class AdvancedMediaPlayer {
    func playVlc(fileName: String) {
        print("Playing vlc file. Name: \(fileName)")
    }

    func playMp4(fileName: String) {
        print("Playing mp4 file. Name: \(fileName)")
    }
}

// 简单音乐播放器的接口
protocol MediaPlayer {
    func play(audioType: String, fileName: String)
}

// 对象适配器
class MediaAdapter: MediaPlayer {
    private var advancedMediaPlayer: AdvancedMediaPlayer

    init(advancedMediaPlayer: AdvancedMediaPlayer) {
        self.advancedMediaPlayer = advancedMediaPlayer
    }

    func play(audioType: String, fileName: String) {
        if audioType == "vlc" {
            advancedMediaPlayer.playVlc(fileName: fileName)
        } else if audioType == "mp4" {
            advancedMediaPlayer.playMp4(fileName: fileName)
        }
    }
}

// 简单音乐播放器
class AudioPlayer: MediaPlayer {
    private var mediaAdapter: MediaAdapter?

    func play(audioType: String, fileName: String) {
        if audioType == "mp3" {
            print("Playing mp3 file. Name: \(fileName)")
        } else if audioType == "vlc" || audioType == "mp4" {
            mediaAdapter = MediaAdapter(advancedMediaPlayer: AdvancedMediaPlayer())
            mediaAdapter?.play(audioType: audioType, fileName: fileName)
        } else {
            print("Invalid media. \(audioType) format not supported")
        }
    }
}

2. 桥接模式（Bridge Pattern）

桥接模式（Bridge Pattern）是一种结构型设计模式，用于将抽象部分与实现部分分离，使它们可以独立地变化，从而降低它们之间的耦合度。

桥接模式常用于需要多个不同平台、不同类型的类之间协同工作的场景，它能够通过将这些类的不同维度分离来实现这个目的，从而使得这些类可以在系统中独立地变化。

具体实现上，桥接模式通过在抽象层和实现层之间增加一个桥接接口（Bridge），来将抽象层和实现层分离。这个桥接接口通常包含了一些基本操作，而抽象层和实现层分别继承这个桥接接口，从而能够相互协作。

以下是一个使用桥接模式的 Swift 5 代码示例，其中将一个图形对象的绘制行为和图形类型进行分离：

  • Swift Code

    // 实现层
protocol Drawer {
    func draw(x: Double, y: Double, radius: Double)
}

class CircleDrawer: Drawer {
    func draw(x: Double, y: Double, radius: Double) {
        print("Draw Circle at (\(x), \(y)) with radius \(radius)")
    }
}

class SquareDrawer: Drawer {
    func draw(x: Double, y: Double, radius: Double) {
        print("Draw Square at (\(x), \(y)) with radius \(radius)")
    }
}

// 抽象层
class Shape {
    let x: Double
    let y: Double
    let radius: Double
    let drawer: Drawer

    init(x: Double, y: Double, radius: Double, drawer: Drawer) {
        self.x = x
        self.y = y
        self.radius = radius
        self.drawer = drawer
    }

    func draw() {
        drawer.draw(x: x, y: y, radius: radius)
    }
}

class Circle: Shape {
    convenience init(x: Double, y: Double, radius: Double) {
        self.init(x: x, y: y, radius: radius, drawer: CircleDrawer())
    }
}

class Square: Shape {
    convenience init(x: Double, y: Double, radius: Double) {
        self.init(x: x, y: y, radius: radius, drawer: SquareDrawer())
    }
}

let circle = Circle(x: 10, y: 20, radius: 5)
let square = Square(x: 30, y: 40, radius: 5)

circle.draw()
square.draw()

在上面的示例中，**Drawer** 接口定义了绘制图形的基本操作，**CircleDrawer** 和 SquareDrawer 则是具体的绘制实现。**Shape** 类则代表图形对象，它包含了一个 Drawer 类型的属性，用于绘制该图形对象。**Circle** 和 Square 类是具体的图形类型，它们继承自 Shape 类，并在初始化时传入不同的绘制实现。

最后当需要创建多个平行的类层次结构，其中一个是实现平台相关代码，另一个是提供跨平台的通用接口，而且这些类层次结构之间的联系需要在运行时建立时，就可以考虑使用桥接模式。

桥接模式的核心思想是将抽象部分与其实现部分分离，以便它们可以独立变化。它允许抽象部分与其实现部分独立扩展，从而提高了系统的可扩展性和灵活性。在桥接模式中，抽象部分通常是指一个抽象类或接口，实现部分通常是指一个具体类。

举个例子，假设我们要设计一个跨平台的绘图软件，支持在不同的操作系统上运行，并且支持不同的绘图形式。我们可以使用桥接模式来设计这个软件，将绘图形式和操作系统分别作为抽象和实现，从而允许在运行时动态地组合不同的抽象和实现。

以下是一个简单的使用桥接模式的 Swift 5 代码示例：

  • Swift Code

    protocol DrawingAPI {
    func drawCircle(x: Int, y: Int, radius: Int)
}

class DrawingAPI1: DrawingAPI {
    func drawCircle(x: Int, y: Int, radius: Int) {
        print("API1.circle at (\(x):\(y)) radius \(radius)")
    }
}

class DrawingAPI2: DrawingAPI {
    func drawCircle(x: Int, y: Int, radius: Int) {
        print("API2.circle at (\(x):\(y)) radius \(radius)")
    }
}

class Shape {
    var drawingAPI: DrawingAPI?

    init(drawingAPI: DrawingAPI) {
        self.drawingAPI = drawingAPI
    }

    func draw() {
        preconditionFailure("This method must be overridden")
    }
}

class CircleShape: Shape {
    var x: Int
    var y: Int
    var radius: Int

    init(x: Int, y: Int, radius: Int, drawingAPI: DrawingAPI) {
        self.x = x
        self.y = y
        self.radius = radius
        super.init(drawingAPI: drawingAPI)
    }

    override func draw() {
        drawingAPI?.drawCircle(x: x, y: y, radius: radius)
    }
}

let shapes = [
    CircleShape(x: 1, y: 2, radius: 3, drawingAPI: DrawingAPI1()),
    CircleShape(x: 5, y: 7, radius: 11, drawingAPI: DrawingAPI2()),
]

for shape in shapes {
    shape.draw()
}

在上面的代码中，我们首先定义了一个名为 DrawingAPI 的抽象类或接口，该类包含了一个名为 drawCircle 的抽象方法。然后，我们分别实现了 DrawingAPI1 和 DrawingAPI2 这两个具体类，它们分别表示不同的绘图 API。

3. 组合模式（Composite Pattern）

组合模式（Composite Pattern）是一种结构型设计模式，它允许你将对象组合成树形结构，以表示”部分-整体”的层次关系。组合模式可以让客户端统一处理单个对象和组合对象，从而简化代码。

组合模式通常用于以下场景：

  • 需要表示对象的层次结构，且每个对象都具有相同的接口。
  • 需要统一处理单个对象和组合对象，而不需要显式区分它们。
  • 需要对对象进行递归组合，从而能够处理复杂的树形结构。

下面是使用 Swift 5 实现组合模式的示例代码：

  • Swift Code

    protocol Component {
    var name: String { get }
    func add(component: Component)
    func remove(component: Component)
    func display(depth: Int)
}

class Leaf: Component {
    private let _name: String
    var name: String {
        return _name
    }

    init(name: String) {
        _name = name
    }

    func add(component: Component) {
        // Leaf 类型不能添加子节点，因此不需要实现该方法
    }

    func remove(component: Component) {
        // Leaf 类型不能删除子节点，因此不需要实现该方法
    }

    func display(depth: Int) {
        print(String(repeating: "-", count: depth) + " " + name)
    }
}

class Composite: Component {
    private let _name: String
    private var _children = [Component]()

    var name: String {
        return _name
    }

    init(name: String) {
        _name = name
    }

    func add(component: Component) {
        _children.append(component)
    }

    func remove(component: Component) {
        if let index = _children.firstIndex(where: { $0 === component }) {
            _children.remove(at: index)
        }
    }

    func display(depth: Int) {
        print(String(repeating: "-", count: depth) + " " + name)
        _children.forEach { $0.display(depth: depth + 2) }
    }
}

// 使用示例
let root = Composite(name: "root")
root.add(component: Leaf(name: "Leaf A"))
root.add(component: Leaf(name: "Leaf B"))

let compositeX = Composite(name: "Composite X")
compositeX.add(component: Leaf(name: "Leaf XA"))
compositeX.add(component: Leaf(name: "Leaf XB"))
root.add(component: compositeX)

let compositeY = Composite(name: "Composite Y")
compositeY.add(component: Leaf(name: "Leaf YA"))
compositeY.add(component: Leaf(name: "Leaf YB"))
compositeX.add(component: compositeY)

root.display(depth: 1)

在这个示例中，**Component** 是组合模式的抽象基类，包含了所有叶子节点和组合节点都具有的方法和属性。**Leaf** 是叶子节点的具体实现，它不能添加子节点。**Composite** 是组合节点的具体实现，它可以添加和删除子节点，并递归调用子节点的 display 方法。

4. 装饰模式（Decorator Pattern）

装饰模式（Decorator Pattern）是一种结构型设计模式，它允许你在不改变对象自身的基础上动态地给对象添加新的行为。它通常被用来解决在不改变现有代码的情况下，动态地给对象添加新的功能或行为。

装饰模式的核心思想是在运行时通过一系列的装饰器包裹原始对象来实现功能的添加。每个装饰器都与被包裹的对象实现相同的接口，因此可以在不修改原始对象的情况下添加新的行为。

下面是一个使用 Swift 5 实现装饰模式的例子，假设我们有一个 Beverage 协议表示饮料，我们想要给饮料添加调料的功能，我们可以使用装饰模式来实现：

  • Swift Code

    // 饮料协议
protocol Beverage {
    var description: String { get }
    var cost: Double { get }
}

// 浓缩咖啡类
class Espresso: Beverage {
    var description: String {
        return "Espresso"
    }
    var cost: Double {
        return 1.99
    }
}

// 调料装饰器基类
class CondimentDecorator: Beverage {
    let beverage: Beverage
    var description: String {
        return beverage.description
    }
    var cost: Double {
        return beverage.cost
    }
    init(_ beverage: Beverage) {
        self.beverage = beverage
    }
}

// 摩卡调料装饰器
class Mocha: CondimentDecorator {
    override var description: String {
        return "\(beverage.description), Mocha"
    }
    override var cost: Double {
        return 0.20 + beverage.cost
    }
}

// 奶泡调料装饰器
class Whip: CondimentDecorator {
    override var description: String {
        return "\(beverage.description), Whip"
    }
    override var cost: Double {
        return 0.10 + beverage.cost
    }
}

在上面的例子中，**Beverage** 协议定义了一个饮料的接口，其中包含一个 description 属性表示饮料的描述和一个 cost 属性表示饮料的价格。**Espresso** 类是一种饮料，实现了 Beverage 协议。

CondimentDecorator 类是装饰器的基类，实现了 Beverage 协议，它包含一个指向被装饰的饮料的引用，并且提供了默认的 description 和 cost 实现。**Mocha** 和 Whip 类是具体的装饰器类，它们继承自 CondimentDecorator 并且实现了自己的 description 和 cost 方法来添加新的行为。

使用装饰模式，我们可以像下面这样使用装饰器来装饰原始的饮料对象：

let beverage: Beverage = Espresso()
//现在我们已经创建了一个 Espresso 饮料对象，接下来我们可以使用 Mocha 和 Whip 装饰器来添加新的行为：
let mocha: Beverage = Mocha(beverage)
print("\(mocha.description) $\(mocha.cost)")

let whip: Beverage = Whip(mocha)
print("\(whip.description) $\(whip.cost)")

在上面的代码中，我们先用 Mocha 装饰器包裹了原始的 Espresso 对象，然后使用 Whip 装饰器包裹了 Mocha 对象。最终，我们得到了一个包含 Espresso、Mocha 和 Whip 的新的饮料对象，并且可以输出其描述和价格。

装饰器模式的优点在于它允许你在不改变原始对象的情况下，动态地添加新的行为。这使得代码更加灵活，因为你可以通过组合不同的装饰器来创建各种不同的对象。另外，装饰器模式还遵循开闭原则，因为你可以添加新的装饰器而不必修改现有的代码。

当然，装饰器模式并不适用于所有情况。如果你需要对对象进行大量的装饰操作，可能会导致代码变得难以维护。此外，在某些情况下，可能更好的解决方案是使用继承来扩展对象的功能，而不是使用装饰器模式。

5. 外观模式（Facade Pattern）

外观模式（Facade Pattern）是一种设计模式，它提供了一个简单的接口，用于隐藏复杂系统的复杂性。它的主要目的是为客户端代码提供一个简单的接口，使其可以访问复杂系统的功能，而不需要了解系统的内部实现细节。

外观模式通常用于以下情况：

  • 当一个系统的复杂性使得它的实现变得困难时，可以使用外观模式来简化接口。
  • 当需要将一个复杂系统划分为更小的子系统时，可以使用外观模式来创建更简单的接口。

以下是一个使用 Swift 5 实现外观模式的示例：

// 复杂系统的子系统
class SubsystemA {
    func operationA() -> String {
        return "Subsystem A operation"
    }
}

class SubsystemB {
    func operationB() -> String {
        return "Subsystem B operation"
    }
}

class SubsystemC {
    func operationC() -> String {
        return "Subsystem C operation"
    }
}

// 定义外观接口
class Facade {
    private let subsystemA = SubsystemA()
    private let subsystemB = SubsystemB()
    private let subsystemC = SubsystemC()

    func operation() -> String {
        var result = "Facade initializes subsystems:\n"
        result += subsystemA.operationA()
        result += subsystemB.operationB()
        result += subsystemC.operationC()
        return result
    }
}

// 客户端代码使用外观接口
let facade = Facade()
print(facade.operation())

在这个示例中，**SubsystemA、SubsystemB** 和 SubsystemC 是一个复杂系统的三个子系统。**Facade** 提供了一个简单的接口来初始化这些子系统，并在客户端代码中使用。客户端代码只需要使用 Facade 对象的 operation() 方法来调用子系统的操作，而不需要了解每个子系统的内部实现细节。

6. 享元模式（Flyweight Pattern）

享元模式（Flyweight Pattern）是一种设计模式，它通过共享相似对象来最小化内存使用和提高性能。这个模式适用于需要创建大量对象的情况，但又不想消耗过多内存的场景。

具体来说，享元模式的核心思想是将对象分为两部分：内部状态（Intrinsic State）和外部状态（Extrinsic State）。其中，内部状态是可以共享的，外部状态是根据需要变化的。这样，在创建对象时，只需创建一部分对象并将其共享，可以显著减少内存消耗。

在实际应用中，享元模式经常用于优化大规模数据处理、图形渲染等领域。

下面是使用 Swift 5 实现享元模式的示例代码：

  • Swift Code

    // 定义享元协议
protocol Flyweight {
    func operation(extrinsicState: Int)
}

// 具体享元类
class ConcreteFlyweight: Flyweight {
    private var intrinsicState: String

    init(intrinsicState: String) {
        self.intrinsicState = intrinsicState
    }

    func operation(extrinsicState: Int) {
        print("Intrinsic state: \(intrinsicState), Extrinsic state: \(extrinsicState)")
    }
}

// 享元工厂
class FlyweightFactory {
    private var flyweights = [String: Flyweight]()

    func getFlyweight(intrinsicState: String) -> Flyweight {
        if let flyweight = flyweights[intrinsicState] {
            return flyweight
        } else {
            let flyweight = ConcreteFlyweight(intrinsicState: intrinsicState)
            flyweights[intrinsicState] = flyweight
            return flyweight
        }
    }
}

// 使用享元模式
let factory = FlyweightFactory()
let flyweight1 = factory.getFlyweight(intrinsicState: "A")
flyweight1.operation(extrinsicState: 1) // Intrinsic state: A, Extrinsic state: 1

let flyweight2 = factory.getFlyweight(intrinsicState: "B")
flyweight2.operation(extrinsicState: 2) // Intrinsic state: B, Extrinsic state: 2

let flyweight3 = factory.getFlyweight(intrinsicState: "A")
flyweight3.operation(extrinsicState: 3) // Intrinsic state: A, Extrinsic state: 3

在上面的示例中，我们定义了一个 Flyweight 协议，具体享元类 ConcreteFlyweight 实现了这个协议。**FlyweightFactory** 是享元工厂类，用于创建和管理享元对象。在使用时，我们可以通过工厂类获取享元对象，并在其中设置外部状态。由于相同的内部状态只会被创建一次，因此 flyweight1 和 flyweight3 实际上是同一个对象，可以共享使用。

7. 代理模式（Proxy Pattern）

代理模式是一种结构型设计模式，它允许你提供一个代理对象来控制对另一个对象的访问。

代理对象充当原始对象的接口，它可以处理客户端的请求并在必要时将请求传递给原始对象。代理模式常用于以下场景：

  • 远程代理：允许客户端通过网络访问远程对象，例如远程服务API。
  • 虚拟代理：在需要时创建开销很大的对象，例如一个大型图片或者视频对象。
  • 保护代理：控制对敏感对象的访问，例如在用户没有权限的情况下禁止修改数据。

下面是一个使用 Swift 5 代码实现虚拟代理的示例：

  • Swift Code

    protocol Image {
    func display()
}

class RealImage: Image {
    private var fileName: String

    init(fileName: String) {
        self.fileName = fileName
        loadFromDisk()
    }

    func display() {
        print("Displaying \(fileName)")
    }

    private func loadFromDisk() {
        print("Loading \(fileName) from disk")
    }
}

class ImageProxy: Image {
    private var realImage: RealImage?
    private var fileName: String

    init(fileName: String) {
        self.fileName = fileName
    }

    func display() {
        if realImage == nil {
            realImage = RealImage(fileName: fileName)
        }
        realImage?.display()
    }
}

// Usage
let image = ImageProxy(fileName: "test.jpg")
image.display()

在这个示例中，**RealImage** 类表示一个真实的图片对象，它负责加载和显示图片。**ImageProxy** 类则是一个代理，它代表了一个 RealImage 对象并控制了对它的访问。当 ImageProxy 对象的 display 方法被调用时，它会先检查 realImage 是否已经存在。如果 realImage 不存在，那么它会创建一个 RealImage 对象并将其存储在 realImage 中，然后调用 display 方法。如果 realImage 已经存在，那么它将直接调用 realImage 的 display 方法。

这个示例展示了代理模式如何允许你使用代理对象来控制对另一个对象的访问。在这个示例中，**ImageProxy** 充当了 RealImage 的接口，并在必要时创建了一个 RealImage 对象。这使得客户端无需知道 RealImage 的实际细节，从而简化了系统的设计。
